# Moonshine Image 产品迭代计划

## 文档定位

本文档是 Moonshine Image 下一阶段的产品与工程迭代路线图。它基于当前会话中已经完成的启动体验、后端迁移、SLBR 接入、界面样式统一、终端日志、主题配置和布局修复重新整理。

Moonshine Image 的核心方向保持不变：

> 打开软件 -> 导入素材 -> 标记要消失的内容 -> AI 处理 -> 对比结果 -> 导出成品。

后续迭代不应把 Moonshine Image 做成通用图片/视频编辑器，而应继续收敛为一个结果明确、首次体验顺滑、长任务可靠、模型能力可扩展的 AI 去除工具。

## 当前产品基线

### 已落地的体验基线

- 启动阶段支持播放透明背景 WebM 启动动画。
- 启动画面结束后静默进入主界面，AI 引擎准备状态通过运行按钮禁用和 tooltip 表达。
- 用户可在全局设置中控制“开始动画”和“静默启动”。
- 空工作区仍保留免责声明，支持首屏拖拽导入图片/视频。
- 图片处理作用范围已移除“仅当前文件”，默认“仅选中文件”，无选中图片时回退当前图片。
- 图片/视频处理按钮、打开目录、试跑样片、关键帧等按钮样式已开始统一。
- 暗色模式下主要按钮视觉优先使用 secondary，图片处理底栏和视频处理主背景使用 primary。
- 全局设置弹窗已改为固定标题/选项卡/底部操作区，仅具体内容滚动。
- 全局品牌色可配置，当前主题方向为薰衣草蓝紫、淡蓝紫、香槟金、翡翠绿、冷宝石红、琥珀金。
- 根页面滚动条已收敛，避免最大化窗口下 Windows 任务栏触发滚动条挤压底栏。

### 已落地的后端基线

- 后端目录已从 IOPaint 迁移为 `server`，内部包命名为 `moonshine_server`。
- 保留擦除模型和未来可能复用的插件能力，移除不再需要的 Gradio 和生成式模型方向依赖。
- 前端按上次全局后端配置进行默认启动。
- 旧配置会在加载时合并迁移：相同配置保留，不兼容项重置为新版本默认格式。
- 运行时目录命名向 `Moonshine-Image` 收敛，并对旧版 `moonshine-client` 运行时目录给出迁移提示。
- 后端管理终端支持 UTF-8 输出、进度类日志合并刷新、0.5 秒节流同步和最大日志行数滚动截断。
- Windows 打包运行时构建已避免 `conda run python -c` 多行参数问题，改为脚本文件方式执行。

### 已落地的模型基线

- 前端模型选择器已支持动态模型列表。
- 后端已提供 Moonshine 模型清单接口。
- Lama 保留原有蒙版擦除流程。
- SLBR 已作为图片透明水印去除模型接入，采用懒加载，避免拖慢启动。
- SLBR 支持当前图片、选中图片、文件夹批处理。
- SLBR 不要求蒙版，切换后隐藏绘制工具和蒙版层。
- SLBR 支持 `tile_size`、`tile_batch` 参数，其中 `tile_batch` 为 1-32 正整数输入。
- CUDA 状态接口已修正显存统计，并可返回 SLBR 推荐参数。

## 产品原则

### 结果优先

用户关心的是“处理后的干净画面”，不是后端、模型名、Python 环境或技术栈。所有交互都应围绕结果闭环设计：

- 能否快速导入。
- 能否准确指定要去除的内容。
- 能否稳定处理。
- 能否清楚对比。
- 能否顺利导出。

### 复杂度后置

默认路径应该简单。模型管理、运行时诊断、参数微调、批处理细节可以存在，但不应打断新用户首次成功。

### 后端可替换

Moonshine 后端不再被视为 IOPaint 的微改，而是项目自身的 `moonshine_server`。未来所有模型、下载、运行时、诊断和任务系统都应围绕这个后端服务演进。

### UI 一致性

前端优先使用 Quasar 原生组件和当前样式体系：

- 按钮使用统一高度、圆角、icon+文本间距和响应式换行逻辑。
- 暗色模式必须同步检查卡片、弹层、下拉菜单、输入框、滚动条和按钮。
- 避免使用全局布局 hack 解决局部问题。
- 页面级滚动、弹窗内部滚动、抽屉滚动需要边界清晰。

## 下一阶段最高优先级

### 1. 模型下载与模型清单体系

#### 目标

把当前“模型已存在才可运行”的状态，推进到“用户可在软件内发现、下载、校验、启用模型”。

#### 范围

- 在项目中建立模型清单文件。
- 每个模型优先使用你提供的 Hugging Face 下载链接。
- 每个模型保留官方备用链接。
- 模型项结构继续沿用并扩展：
  - `id`
  - `label`
  - `description`
  - `type`
  - `installed`
  - `requiresMask`
  - `downloadable`
  - `sourceLinks`
  - `files`
  - `size`
  - `sha256`
  - `recommendedDevice`
  - `minimumVram`
  - `parameters`
- 后端提供模型列表、下载、校验、安装状态刷新接口。
- 前端模型选择器显示“已安装 / 可下载 / 不适合当前设备 / 缺失文件”等状态。
- 默认发布包仍优先携带最核心模型，其他模型可按需下载。

#### 暂不做

- 不在下一轮强行做完整模型商店式页面。
- 不引入账号体系、云端授权或复杂推荐系统。

#### 验收标准

- 删除某个可下载模型后，软件能识别缺失并显示下载入口。
- 下载完成后无需重启即可刷新模型状态。
- 模型文件损坏时能提示重新下载，而不是运行时崩溃。
- 无网络或下载失败时能保留当前素材状态。

### 2. SLBR 图片能力打磨

#### 目标

把 SLBR 从“可运行”提升为“可信赖的透明水印去除能力”。

#### 范围

- 固化当前新训练模型作为默认 SLBR checkpoint。
- 优化不同显存设备的推荐参数。
- 对大图 tiled inference 做稳定性测试。
- 增加处理耗时和失败原因的简明提示。
- 文件夹批处理输出目录结构保持清晰。
- 保留 debug 能力，但默认只向用户展示 clean 结果。

#### 验收标准

- 当前图片、选中图片、文件夹批处理均稳定可用。
- 低显存设备不会因为默认参数过高直接 OOM。
- 结果加入图片历史后，可对比、撤销、下载。

### 3. 视频处理可靠性

#### 目标

视频用户可以接受慢，但不能接受不可预期。下一阶段需要让视频任务具备可预览、可恢复、可诊断的安全感。

#### 范围

- 继续完善 3 秒 / 10 秒样片试跑。
- 处理完成提示统一为“可点击打开目录查看视频文件”。
- 阶段进度清晰表达：
  - 拆帧
  - 处理帧
  - 合成视频
  - 写出结果
  - 清理临时文件
- 失败时保留最近失败现场。
- 支持清理旧临时目录，但不清理正在处理或最近失败任务。
- 优化视频长任务日志，避免进度条刷屏。

#### 后续增强

- 任务队列。
- 断点续跑。
- 失败批次重试。
- 预计剩余时间。
- 视频透明水印一致性处理。

## 中期路线

### 1. 图片智能选择

#### 方向

引入 SAM / SAM2 / MobileSAM 类模型，但产品文案不直接强调技术名，统一包装为“智能选择”。

#### 目标

- 点一下选择要去除的目标。
- 支持正向点、反向点、框选辅助。
- 自动生成蒙版后仍允许手动修正。
- 与现有绘制工具共享蒙版数据结构。

#### 优先级判断

该方向提升的是输入效率，不是结果质量本身。因此应排在透明水印结果质量和视频可靠性之后。

### 2. OCR 文字水印流程

#### 方向

OCR 不应作为独立文字识别工具展示，而应绑定“去文字 / 去字幕 / 去水印”任务。

#### 目标

- 自动识别图片中文字区域。
- 用户可选择识别结果生成蒙版。
- 对字幕类视频场景预留批量识别能力。
- 仍允许手动修正识别区域。

### 3. 更多擦除模型

#### 方向

保留 Lama、SLBR，并逐步接入适合不同场景的擦除模型。

#### 候选维度

- 通用物体移除。
- 文字/字幕移除。
- 动漫/插画场景。
- 高清图修复。
- 低显存快速模型。

#### 要求

- 每个模型必须明确适用场景，而不是只展示模型名。
- 不适合当前设备的模型不应默认可运行。
- 下载、校验、加载失败要有统一错误结构。

## 后端工程路线

### 1. Moonshine Server 服务边界

#### 目标

让 `server/moonshine_server` 成为稳定、可维护、可扩展的后端服务，而不是前端临时调用脚本集合。

#### 计划

- 保持 `/api/v1/batch_inpaint` 兼容 Lama 旧流程。
- Moonshine 自有模型统一走通用图片模型处理接口。
- 区分需要蒙版模型和无蒙版模型。
- 抽象模型注册、模型检查、模型下载、模型加载、模型卸载。
- 将视频处理任务逐步从同步请求改为任务式 API。

### 2. 运行时与打包

#### 目标

降低用户设备差异对首次启动的影响。

#### 计划

- 继续使用可复用的 Windows 离线运行时 artifact。
- 构建时保留 torch / torchvision 版本约束，避免重复安装已存在的大包。
- 移除无用运行时依赖，尤其是 Gradio、Diffusers、Accelerate、ControlNet 类不再需要的生成模型依赖。
- 对 CUDA 版本、torch 版本、显存、模型文件进行启动前检查。
- 后端项目结构检测失败时继续提供旧版运行时目录删除提示。

### 3. 诊断体验

#### 目标

普通用户只看到一句可理解错误，高级用户可以打开诊断。

#### 计划

- 运行按钮失败态提示“打开诊断”。
- 后端管理页保留完整日志，但默认不弹复杂日志。
- 错误分类继续收敛：
  - AI 引擎未启动
  - 模型未找到
  - 运行时不可用
  - CUDA 不可用
  - 请求未响应
  - 响应结果无法识别
  - 输出目录不可写

## 前端体验路线

### 1. 主题和暗色模式

#### 目标

让主题配置成为稳定能力，而不是临时换色。

#### 计划

- 全局设置中继续暴露品牌色配置。
- 统一检查所有弹层、菜单、抽屉、输入框、按钮、卡片的暗色模式。
- 暗色模式下避免浅灰背景配白字。
- 品牌色变更后，图片页、视频页、弹窗、按钮同步响应。
- 后续可考虑预设主题：
  - Moonshine 默认
  - 高对比暗色
  - 低饱和护眼

### 2. 按钮和控件统一

#### 目标

所有高频操作按钮形成一致视觉语言。

#### 计划

- 图片处理、视频处理、全局设置使用统一 action button 风格。
- 长按钮 icon 和文本作为整体居中，不强行拆分左右区域。
- 下拉框、数字输入、开关优先使用 Quasar 原生组件。
- 页面变窄时按钮不应出现挤压、截断和不自然换行。

### 3. 布局边界

#### 目标

避免再出现全局设置滚动、根页面滚动、底栏被挤压等问题。

#### 计划

- 根页面不产生浏览器级滚动条。
- 页面内部滚动只出现在明确容器内。
- 弹窗标题、选项卡、底部操作固定，内容区滚动。
- 抽屉、底栏、浮动绘制工具遵循 Quasar layout 规则。
- 避免用 `100vw` 撑开根布局，优先使用 `width: 100%`、flex、min-width: 0。

## 暂缓方向

以下方向暂缓，除非它们直接提升去除结果或主流程可靠性：

- 通用裁剪、旋转、调色。
- 与去除流程无关的图片编辑工具。
- 复杂素材库管理。
- 云端账号、社区、模板市场。
- 生成式图片创作能力。
- 视频非去除类剪辑功能。

## 分阶段路线图

### 近期版本

目标：让模型能力可下载、可校验、可启用。

- 建立模型清单文件。
- 接入 Hugging Face 首选链接和官方备用链接。
- 完成模型下载、校验、刷新状态接口。
- 前端展示已安装、可下载、缺失、设备不适配状态。
- 打磨 SLBR 图片透明水印结果和推荐参数。
- 继续检查暗色模式和按钮统一遗漏项。

### 下一版本

目标：让视频处理变得更可信。

- 完善视频样片试跑闭环。
- 增加阶段进度和失败现场恢复。
- 处理日志进一步结构化。
- 优化输出目录、替换源文件、打开目录等完成态体验。
- 为视频任务队列和断点续跑做后端 API 准备。

### 中期版本

目标：提升输入效率和模型覆盖面。

- 接入智能选择模型。
- 接入 OCR 辅助文字水印识别。
- 增加更多擦除模型。
- 完善模型管理 UI。
- 根据设备性能推荐模型和参数。

### 长期版本

目标：形成稳定的 AI 去除工作台。

- 图片和视频任务统一队列。
- 长任务断点续跑。
- 批处理预设。
- 模型下载镜像和完整校验体系。
- 面向开源贡献者的后端扩展文档。

## 成功标准

一个版本是否值得发布，可以用以下标准判断：

- 不懂 Python 的 Windows 用户能否在 3 分钟内完成第一张图片处理。
- 用户是否可以不打开后端管理页完成一次成功处理。
- 图片透明水印结果是否足够稳定，值得截图传播。
- 视频处理是否能让用户知道当前进度、输出在哪里、失败后如何恢复。
- 模型下载是否可理解、可恢复、可校验。
- 暗色模式和主题配置是否没有明显割裂。
- 核心流程是否仍然简单。

## 一句话总结

Moonshine Image 下一阶段的重点不是扩展成更复杂的编辑器，而是把“AI 去除”做成一个用户可信赖的完整产品：

> 默认顺滑启动，模型按需下载，图片结果可靠，视频任务可预期，失败可以诊断，导出路径清晰。
